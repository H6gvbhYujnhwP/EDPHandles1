# Staged Pathway for Bringing Cosma Content into the EDP Handles Website

## Recommended Approach

The safest and most effective route is **not** to try to copy the whole Cosma website in one pass. Instead, we should treat this as a **structured content migration and redesign adaptation**. The aim would be to bring selected Cosma product images, range descriptions, finish information, and supporting resources into the EDP Handles site while keeping the **same premium editorial style, dark showroom aesthetic, typography system, and visual language** that we have already established.

That means we should separate the work into two streams. The first stream is **content capture and organisation**. The second is **page-by-page implementation** into the EDP site. Keeping those separate will reduce mistakes, prevent messy duplication, and make it much easier for you to review progress in small stages.

## What We Need To Do Before Copying Images and Descriptions

Before we bring over product images and descriptions, we need to define exactly **what we are allowed to use, what we want to curate, and how it should be structured for the UK market**. Cosma’s website is broader and more catalogue-led than the current EDP Handles site, so if we copy content directly without a framework, the result will feel inconsistent and overloaded.

The first practical requirement is to create a **content inventory**. This means listing every page, range, image set, description block, finish group, downloadable catalogue, and trade message that might be relevant. Once that inventory exists, we can decide what should appear on EDP Handles immediately, what should wait until later, and what should be rewritten to fit the UK distributor positioning rather than copied literally.

The second requirement is to define a **content model** for the EDP site. For example, every showcased range should eventually have a consistent structure such as: **range name**, **category**, **short design-led description**, **technical summary**, **materials**, **available finishes**, **hero image**, **gallery images**, and **trade enquiry CTA**. If we do that first, importing content becomes systematic instead of improvised.

## How To Preserve the Existing Design Style

We should keep the current EDP visual identity as the master design system and then pour the new content into it. In other words, the Cosma content should be **adapted into the EDP experience**, not the other way around.

The Nano Banana images should continue to be used for the most prominent brand-led moments such as hero banners, atmospheric page openers, and major editorial transitions. Real Cosma product images should then be used in a more structured way inside product grids, range pages, finish references, and gallery panels. That gives us the best of both worlds: the site still feels crafted and premium, but it becomes commercially useful and product-specific.

A good design rule for the next phase is this: **Nano Banana for brand atmosphere, original product imagery for catalogue credibility**. That will preserve the same style while preventing the site from feeling either too generic or too technical.

## Recommended Site Structure

Based on the Cosma review, the next iteration of EDP Handles should be structured as a curated multi-page distributor site.

| Stage structure | Purpose | Notes |
|---|---|---|
| Home | Brand-led entry page | Keep the current style and refine links into real content pages |
| About EDP Handles | UK distributor positioning | Explain relationship to Italian design supply and trade support |
| Handles | Main product family page | Organise by understandable UK-facing categories |
| Open Systems | Dedicated systems page | Use named ranges such as MIRA, TELA, ERRE 8, EOS, AURA, and CUBO |
| Finishes | Finish library or curated matrix | One of the clearest gaps in the current site |
| Express Collection | Fast-turnaround or stocked offer | Add only if commercially relevant |
| Catalogues & Downloads | PDFs, brochures, literature | Important for designers and trade buyers |
| Trade / Contact | Consultation and enquiry path | Should become stronger and more specific |

This does **not** mean we must build everything at once. It simply means this should be the destination architecture, so each new page fits into a clear framework from the start.

## Best Staged Workflow

The best workflow is to move in **small approval rounds**, with each round containing one clear objective. That way, you can check direction before we scale the build further.

| Stage | What we do | Output |
|---|---|---|
| 1 | Confirm sitemap and page priorities | Final agreed structure before build |
| 2 | Build one product-family page template | A repeatable layout for future pages |
| 3 | Collect and prepare content for that page | Range names, product images, descriptions, finishes |
| 4 | Implement that page in the live site | One finished page in the approved design style |
| 5 | Review and refine | Approve before duplicating pattern to next page |
| 6 | Repeat page by page | Controlled scale-up across the site |

## The Content Migration Process

For each page, we should use the same repeatable process.

First, we select the page scope. For example, if we start with **Handles**, we decide which handle groups will appear there. Second, we collect the relevant Cosma source material: product names, descriptions, images, and any finish information. Third, we rewrite or adapt the copy into **EDP Handles language** so it feels like a UK distributor site rather than a direct mirror of the Italian source. Fourth, we place the content into the approved page layout while preserving the existing visual system.

This is important because not every Cosma description should be copied word for word. Some content is manufacturer-facing, some is catalogue shorthand, and some is better used as supporting technical information rather than homepage copy. We should therefore think in terms of **translation and adaptation**, not raw duplication.

## The Small-Stage Build Order I Recommend

The most sensible sequence is to start with the page that proves the structure, not the most complicated page.

| Recommended order | Why this should come first |
|---|---|
| 1. Handles page | This is likely the core commercial page and will establish the product-page template |
| 2. Finishes page | High-value, structurally important, and easier to review than a full catalogue |
| 3. Open Systems page | Lets us introduce named Cosma ranges in a curated way |
| 4. About EDP Handles page | Strengthens distributor credibility and brand story |
| 5. Catalogues & Downloads page | Adds trade utility once content foundations are ready |
| 6. Express page | Useful only if the offer is confirmed for the UK market |
| 7. Home page refinement | Final pass to connect everything cleanly |

## What This Means Practically for Images and Descriptions

To bring the product images and descriptions into the website properly, we should first create a **working library** outside the live page code. That library would hold the product images, filenames, range names, copied or rewritten descriptions, finish references, and source page URLs. Once that content bank exists, we can feed each page from structured material instead of scraping ad hoc while building.

We should also decide whether the site will show **every product** or only a **curated set of lead ranges**. If the answer is “every product,” the workload becomes much larger and will require a fuller data structure. If the answer is “curated,” we can move faster while keeping the site elegant.

## Recommended First Next Step

The best first step is to **freeze the structure before importing content**. In practical terms, I recommend that we do the following next:

| Immediate next action | Why |
|---|---|
| Approve the target sitemap | Prevents content being imported into the wrong structure |
| Choose the first page to build | Keeps delivery small and reviewable |
| Define the content fields for that page | Makes migration consistent |
| Then collect only the content for that page | Faster, cleaner, and easier to check |

## My Suggested Phase-One Build Decision

If you want to take this in small, safe stages, I recommend that we start with **one new page only: the Handles page**. That page would let us establish the design pattern for product-family pages, import a first controlled set of real Cosma product content, and confirm that the structure works before we roll the same system out to Finishes, Systems, and Downloads.

Once that page is approved, the rest of the build becomes much more predictable.
